# 政策分析静态网站设计文档

日期：2026-04-13

## 1. 项目目标

本项目要建设一个可部署到 GitHub Pages 的政策分析静态网站，用于持续追踪国家、山东省、青岛市、青岛西海岸新区发布的企业相关政策，并判断这些政策是否可能给公司带来补贴、奖励、项目申报、试点示范、政府采购、市场拓展或其他经营机会。

服务对象是一家位于青岛西海岸新区的民营综合型企业，业务覆盖建设施工、物业、安保、商业管理、资产管理、外贸、商砼等方向。

第一版目标不是做复杂后台系统，而是先做一个能上线、能每天更新、能筛选政策、能输出企业适配分析的实用版本。

## 2. 第一版范围

第一版包含以下功能：

- 静态网站首页，展示政策看板、重点机会、政策分类和更新时间。
- 政策库页面，支持按地区、业务板块、匹配等级、是否涉及补贴进行筛选。
- 补贴机会页面，集中展示疑似补贴、奖励、专项资金、申报指南、试点示范等高价值政策。
- 政策详情页或详情弹窗，展示政策基本信息、AI 分析结果、公司匹配判断和下一步建议。
- 数据源说明页面，说明系统追踪哪些政策网站，以及如何通过 GitHub Issue 添加新的政策网址。
- GitHub Issue 表单，用于让用户提交新增政策来源或单条政策链接。
- GitHub Actions 定时任务，每天自动抓取政策来源、去重、分类、调用 DeepSeek API 分析，并更新网站数据文件。
- DeepSeek API 增强分析，用于生成面向公司业务的政策利好判断。

第一版暂不包含以下功能：

- 登录系统。
- 独立数据库后台。
- 微信、短信或邮件自动推送。
- 正式申报材料自动生成。
- 政府网站验证码绕过。
- 法律、财税或申报结果的最终结论。

## 3. 用户画像

主要用户是公司负责人、综合管理人员、财务人员、业务部门负责人或政策申报负责人。

这些用户不一定懂技术，所以网站必须做到：

- 一眼看出今天有没有值得关注的新政策。
- 一眼看出哪些政策可能和公司补贴有关。
- 能按公司业务板块快速筛选。
- 能看到通俗、明确、可执行的下一步建议。
- 不要求用户理解代码或数据结构。

## 4. 主要政策范围

系统重点关注以下层级：

- 国家部委政策。
- 山东省政策。
- 青岛市政策。
- 青岛西海岸新区政策。

重点业务关键词包括：

- 建设施工。
- 物业服务。
- 商砼、预拌混凝土。
- 装配式建筑、绿色建筑、智能建造。
- 房地产。
- 安保服务。
- 商业管理、资产管理。
- 城市更新。
- 社区嵌入式服务。
- 土地、园区、公共服务。
- 外贸、跨境电商、出口信用保险。
- 民营经济、中小企业、专项资金、补贴、奖励、申报指南。

## 5. 推荐技术方案

采用“GitHub Pages 静态网站 + GitHub Actions 每日自动任务 + DeepSeek API 分析”的方案。

原因如下：

- GitHub Pages 适合免费部署静态网站。
- GitHub Actions 可以每天自动运行抓取和分析任务。
- 数据可以保存为 JSON 文件，网站直接读取，不需要第一版引入数据库。
- DeepSeek API 适合中文政策文本分析，成本相对可控。
- 后续如果需要登录、数据库、消息推送，可以在此基础上升级。

## 6. 系统结构

系统分为四个部分：

### 6.1 前端静态网站

职责：

- 展示政策列表。
- 展示重点补贴机会。
- 支持搜索和筛选。
- 展示政策详情和 AI 分析。
- 说明如何新增政策来源。

建议页面：

- 首页：政策雷达总览。
- 补贴机会：A级和疑似补贴政策。
- 政策库：全部政策列表。
- 数据源：已追踪网站和新增方式说明。
- 关于公司画像：说明系统按什么公司背景进行匹配分析。

### 6.2 数据文件

第一版使用静态 JSON 文件保存数据。

建议数据文件：

- `data/policies.json`：政策列表和分析结果。
- `data/sources.json`：固定政策来源。
- `data/company-profile.json`：公司业务画像。
- `data/run-log.json`：最近一次自动更新结果。

这样做的好处是结构简单，GitHub Pages 可以直接读取。

### 6.3 自动更新任务

GitHub Actions 每天定时运行。

主要流程：

1. 读取固定政策来源。
2. 读取 GitHub Issue 中用户提交的政策网址。
3. 抓取政策网页标题、发布时间、发布单位、正文和原文链接。
4. 对政策进行去重。
5. 根据地区、业务板块、关键词进行初步分类。
6. 调用 DeepSeek API 生成政策分析。
7. 生成或更新 `data/policies.json`。
8. 自动提交更新后的数据到仓库。
9. GitHub Pages 展示最新数据。

### 6.4 AI 分析模块

DeepSeek API 的分析重点不是简单总结政策，而是回答以下问题：

- 这条政策和公司哪些业务有关？
- 是否涉及补贴、奖励、专项资金、申报、试点或政府采购机会？
- 公司是否可能符合条件？
- 还需要确认哪些信息？
- 是否建议重点跟进？
- 下一步应该准备什么材料或联系哪个主管部门？

AI 输出必须保守，不得编造补贴金额、申报截止日期或政策条件。

## 7. 政策匹配等级

每条政策需要给出匹配等级：

- A级：重点跟进。明确涉及公司业务，且可能存在补贴、申报、奖励、试点、政府采购或项目机会。
- B级：持续关注。与公司业务有关，但暂时没有明确申报入口或补贴细则。
- C级：低优先级。关联较弱，短期价值有限。
- D级：暂不适用。明显不适合公司、地区不符、主体不符或政策已过期。

网站默认优先展示 A级和疑似补贴政策。

## 8. 政策详情展示内容

每条政策详情建议包含：

- 政策名称。
- 发布单位。
- 发布时间。
- 适用地区。
- 原文链接。
- 相关业务板块。
- 匹配等级。
- 是否疑似补贴政策。
- 政策核心内容。
- 可能利好。
- 公司是否可能符合。
- 需要进一步确认的信息。
- 建议负责部门。
- 建议准备材料。
- 一句话结论。
- 风险提示。

## 9. GitHub Issue 新增政策来源

为了让用户不用改代码，第一版采用 GitHub Issue 表单新增政策网址。

建议表单字段：

- 政策网址。
- 来源地区：国家、山东省、青岛市、西海岸新区、其他。
- 相关业务：建设施工、物业、安保、外贸、商砼、商业管理、资产管理、城市更新、其他。
- 备注。

自动任务会读取带有指定标签的 Issue，例如 `policy-source`，并将其中的网址纳入下一次抓取。

## 10. 风险与边界

需要明确以下边界：

- 政策抓取可能受到政府网站结构变化、反爬限制、验证码、附件格式等影响。
- AI 分析只能作为初筛和辅助判断，不能替代正式法律、财税、审计或主管部门意见。
- 是否能成功申报补贴，仍需以正式申报指南和主管部门答复为准。
- 对金额、截止日期、申报条件等关键信息，必须保留原文链接供人工复核。
- 第一版不处理需要登录、验证码或复杂动态渲染的网站。

## 11. 第一版验收标准

第一版完成后，应满足以下标准：

- 网站可以在本地运行并展示政策看板。
- 网站可以部署到 GitHub Pages。
- 政策数据来自 JSON 文件，页面可以正常筛选和搜索。
- GitHub Actions 可以每天自动运行。
- 自动任务可以读取固定来源和 GitHub Issue 提交的网址。
- 自动任务可以调用 DeepSeek API，并生成政策分析结果。
- 没有 DeepSeek API Key 时，系统能给出明确错误提示或降级为规则分析。
- README 中说明如何配置 `DEEPSEEK_API_KEY`、如何添加政策来源、如何部署到 GitHub Pages。

## 12. 后续可扩展方向

后续可以逐步增加：

- 微信或邮件推送。
- 政策申报日历。
- 申报材料清单自动生成。
- 公司资质和财务数据匹配。
- 多公司画像切换。
- 数据库后台。
- 登录权限。
- 政策收藏、跟进状态、责任人分配。

## 13. 当前推荐结论

第一版应优先建设一个低成本、可部署、可自动更新的 GitHub Pages 政策分析静态网站。

核心路线是：

静态网站展示结果，GitHub Actions 每天抓取和分析政策，DeepSeek API 生成企业适配分析，GitHub Issue 表单负责新增政策来源。
